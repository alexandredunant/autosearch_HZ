# Autoresearch Hazard – Fully Autonomous Point‑Process Research

This is an experiment to have the LLM perform **completely autonomous** research on hazard prediction using a Bayesian point‑process model. The LLM edits `train.py`, runs experiments, keeps only improvements, and **stops automatically** after a plateau of 20 consecutive non‑improving attempts.

## Research Context

- **Goal:** Maximize `val_loglik` (held‑out log‑likelihood) for a presence‑only point‑process model.
- **Model:** Bayesian point process with Normal(0,1) prior on coefficients, MAP estimation via L‑BFGS‑B.
- **Features:** Static rasters (terrain, land cover, etc.) plus temporal  rolling aggregates of dynamic variables (precipitation, temperature, lightning). A bias raster (`bias_sampling.tif`) is always included as a log‑offset.
- **Data:** Pre‑processed and stored in `prepared_point_data_<hazard>.npz` (generated by running `python prepare.py` once).
- **Metric:** Higher `val_loglik` is better.
- **Each experiment runs on a single GPU.**

## File Roles and Permissions

| File | Purpose | Editable? |
|------|---------|-----------|
| `prepare.py` | Data generation (run once) and immutable data loading. Contains `build_point_process_splits()`. | **NO** – Read‑only context. |
| `train.py` | Model definition and training loop. Contains `STATIC_FEATURE_NAMES` constant. | **YES** – Only the `STATIC_FEATURE_NAMES` list. The rest is off‑limits. |
| `program.md` | This protocol document. | **NO** – Read‑only context. |
| `results.tsv` | Log of all experiments. | Append only; never commit. |
| `run.log` | stdout/stderr of the latest experiment. | Temporary; overwritten each run. |
| `latest_model_summary.txt` | Coefficients and validation metric from the most recent run. | Generated by `train.py`; read for insight. |

## Setup (First Run Only – Perform Once)

You are responsible for the entire setup **without asking for confirmation**. Execute the following steps exactly once:

1. **Determine hazard type:** Read the first few lines of `prepare.py` to find the variable `HAZARD_TYPE` (e.g., `"landslide"`). Use this value in branch naming.
2. **Propose a run tag** based on today's date (e.g., `apr9`). The branch name must be `autoresearch/<hazard>-<tag>`. Create it:
   ```bash
   git checkout -b autoresearch/<hazard>-<tag>
   ```
3. **Generate the data file** if it does not already exist:
   ```bash
   python prepare.py
   ```
   This creates `prepared_point_data_<hazard>.npz`. If the file is already present, skip this step.
4. **Initialize `results.tsv`** with the header row (TAB‑separated):
   ```
   commit	val_loglik	status	description
   ```
   Add `results.tsv` to `.gitignore` (do not commit it).
5. **Proceed immediately** to the baseline experiment. Do not wait for any human input.

## Experimentation

### What You CAN Do

- **Edit `STATIC_FEATURE_NAMES`** in `train.py`. Add or remove **exactly one** feature name per experiment.
  Example: `STATIC_FEATURE_NAMES = ["elevation", "slope"]`
- **Write a descriptive commit message** summarizing the change (e.g., `"add static: precipitation_60d_sum"`).
- **Run the experiment**:
  ```bash
  python train.py > run.log 2>&1
  ```

### What You CANNOT Do

- **Do NOT edit `prepare.py`.** It is immutable context.
- **Do NOT edit any part of `train.py` outside the `EDITABLE CONSTANTS` section.** The model architecture, optimizer, and training loop are fixed.
- **Do NOT install new packages.** Only the packages already present in the environment are allowed.
- **Do NOT change the evaluation metric.** The point‑process log‑likelihood is the ground truth.

### Simplicity Criterion

- **One change per experiment.** Adding two features at once is forbidden. You must isolate the effect of each feature.
- **Lock successful trials** You have to lock features that have provided improvement and cannot removed previous locked features.
- **Small improvements that add complexity are discouraged.** A feature that yields a tiny `val_loglik` gain but is hard to interpret should be weighed against its conceptual cost.
- **Removing a feature and seeing no degradation is a win.** It simplifies the model.

### Baseline Run

The **first experiment** must use an empty feature list:
```python
STATIC_FEATURE_NAMES: list[str] = []
```
Commit message: `"baseline (bias only)"`. Log it with status `keep` (the baseline is always kept).

## Output Format

After `train.py` finishes, it prints exactly one line containing the validation metric:
```
val_loglik: -8430.7653
```
Extract it with:
```bash
grep "^val_loglik:" run.log | awk '{print $2}'
```
If the run crashes, this command returns an empty string.

## Logging Results

After each experiment, append a TAB‑separated line to `results.tsv`:

```
<short-commit-hash>	<val_loglik>	<status>	<description>
```

- **commit:** Short Git hash (7 characters) of the experiment commit.
- **val_loglik:** The extracted value. Use `-1e18` for crashes.
- **status:** `keep`, `discard`, or `crash`.
- **description:** Brief summary (same as commit message).

Example:
```
commit	val_loglik	status	description
a1b2c3d	-8430.7653	keep	baseline (bias only)
b2c3d4e	-8420.1234	keep	add static: elevation
c3d4e5f	-8450.0000	discard	add static: slope (worse)
d4e5f6g	-1e18	crash	add static: missing_feature
```

**Important:** Do **not** commit `results.tsv`. It is an untracked log file.

## The Autonomous Loop with Plateau Stop

**You are fully autonomous. Do not pause, ask questions, or wait for confirmation.** The loop runs until the plateau condition is met.

### Internal State You Must Track

Maintain these two variables in your reasoning (they are not written to a file, just remembered):

- **`best_val_loglik`**: The highest `val_loglik` seen so far. Initialize to `-inf`.
- **`plateau_count`**: Number of consecutive experiments that did **not** improve over `best_val_loglik`. Start at `0`.

### Loop Algorithm

```
best = -inf
plateau = 0
MAX_PLATEAU = 20

while plateau < MAX_PLATEAU:
    1. Decide on ONE change to STATIC_FEATURE_NAMES.
       - You may add a new feature not yet tried.
       - You may remove a feature that showed weak or negative coefficient in `latest_model_summary.txt`.
       - You may re‑add a previously discarded feature if you suspect interaction effects.
    2. Edit train.py accordingly.
    3. Commit with a short descriptive message.
    4. Run: python train.py > run.log 2>&1
    5. Extract val_loglik.
    6. If crash:
         - Inspect tail -n 30 run.log.
         - If the error is trivial (typo, missing feature name), fix it, amend the commit, and re‑run (do not count as plateau).
         - If unfixable, log status "crash", git reset --hard HEAD~1, plateaux += 1, continue.
    7. Compare val_loglik to best.
       - If val_loglik > best:
           best = val_loglik
           plateaux = 0
           log status "keep"
           (Commit remains on branch)
       - Else:
           plateaux += 1
           log status "discard"
           git reset --hard HEAD~1
    8. Append the result line to results.tsv.
    9. (Optional) Print progress: echo "Plateau: $plateau/$MAX_PLATEAU | Best: $best"
```

### Stopping Condition

When `plateau` reaches **20**, you **must stop** the loop and print a final summary:

```
=== PLATEAU REACHED (20 consecutive non‑improving attempts) ===
Best val_loglik: <best>
Total experiments: <count>
Branch: <branch-name>
Final commit: <commit-hash>
```

Then **exit** the autonomous session. Do not make further edits or runs.

## Crash Handling

- **Typo / missing import:** Fix immediately, amend commit, re‑run. Do **not** increment plateau counter.
- **Feature does not exist:** Log as `crash`, increment plateau counter, revert commit.
- **Model convergence failure (e.g., singular matrix):** Treat as crash and move on.

## Ideas When Stuck

- Review `latest_model_summary.txt` for coefficients near zero — these are candidates for removal.
- Check the list of all available static features printed by `prepare.py` (run `python prepare.py` once more if needed to see the list).
- Consider removing the least significant feature even if it slightly hurts performance—it might allow another feature to shine in a later experiment.
- Re‑visit features that were discarded early; with a different feature set they may become beneficial.
